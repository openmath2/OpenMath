# OpenMath 파이프라인 Ablation Study

> 각 컴포넌트가 빠지면 무엇이 망가지는지 — 왜 이 아키텍처여야 하는가에 대한 근거 문서.

---

## 0. 이 문서를 어떻게 읽으면 좋은가

OpenMath 는 "중학교 수학 동형 문제를 LLM 으로 자동 생성하는" 서비스다. 단순히 ChatGPT 에 "이차방정식 문제 3개 만들어줘" 하는 것과 무엇이 다른가? 그 차이를 만들어내는 핵심이 **6단계 검증 파이프라인** 이다.

이 문서는 다음 질문에 답한다:

> "왜 ChatGPT 한 번 호출로 끝내지 않고, 6 단계나 거치는가? 각 단계가 정말 필요한가? 빠뜨리면 무엇이 망가지는가?"

**구성**

- §1 — 사전 지식: OpenMath 가 풀려는 문제와 LLM 단독의 한계
- §2 — 전체 파이프라인 한 눈에
- §3 — 6 단계 각각의 ablation (가장 중요한 부분)
- §4 — "왜 Multi-Agent Debate 가 아니라 결정론 게이트인가" — 대안 아키텍처와의 비교
- §5 — 정량적 검증 실험 설계 (논문/캡스톤 발표용)
- §6 — 결론

## 1. 사전 지식 — OpenMath 가 풀려는 문제

### 1.1 동형 문제(isomorphic problem)란 무엇인가

학원 강사가 학생에게 풀린 문제와 **"같은 평가 목적을 가지면서 표면이 다른"** 문제. 예:

| 원본 문제 | 동형 문제 (구조 동형) | 동형 문제 (개념 동형) |
|---|---|---|
| `2x + 3 = 11` 의 해를 구하시오 | `5x − 2 = 13` 의 해를 구하시오 | `\(2x + 1\) − \(x + 3\) = 0` 의 해를 구하시오 |
| 평가: 일차방정식 풀이 (이항·계수 정리) | 평가: **동일** (숫자만 교체) | 평가: **동일** (식 전개 추가, 같은 기법) |

**왜 어려운가:** 표면을 바꾸기는 쉽지만, **"평가 차원을 보존한 채"** 바꾸기는 어렵다. 예를 들어 위의 `2x + 3 = 11` 을 `2x² + 3 = 11` 로 바꾸면 식별자는 비슷해 보이지만 평가 차원이 **일차 → 이차** 로 완전히 달라진다. 학원 강사가 "일차방정식 숙제 더 풀려야지" 하고 이 문제를 내면 학생의 학습 목표를 벗어난다.

### 1.2 LLM 단독으로는 왜 안 되는가 — MathTrap300 사례

대형 언어 모델은 수학 문제 *생성* 은 잘 하지만 *정답 검증* 은 신뢰할 수 없다. 학계에 잘 알려진 사례:

- **MathTrap300**: 평범한 수학 문제처럼 보이지만 의도적으로 함정을 심은 300 문항 벤치마크. GPT-4 가 자기가 만든 문제의 정답을 자기가 또 풀게 하면 약 **30~40% 가 틀린다.**
- 원인: LLM 의 풀이는 "다음 토큰 예측" 이라 **상징(symbolic) 계산** 이 아니라 **암기·패턴 매칭** 에 가깝다. 1차식은 거의 다 맞지만, 분수·근호·다항식 곱의 정확한 정리는 자주 틀린다.

이 한계는 OpenMath 의 **결정론적 검증** 설계 (D-1) 의 근거가 된다:

> *"LLM 은 생성을, SymPy 는 검증을. 두 역할을 절대 섞지 않는다."*
> — [`docs/specs/architecture.md`](./architecture.md) D-1

### 1.3 평가 차원(evaluation dimensions)이라는 개념

OpenMath 는 "이 문제가 무엇을 평가하는가" 를 4 개 차원으로 정량화한다 ([USER_FLOW.md §3.4](../product/USER_FLOW.md)):

| 차원 | 코드 | 의미 |
|---|---|---|
| **개념 이해** | `concept` | 수학 개념의 의미를 아는가 (예: 일차식의 정의) |
| **절차 숙달** | `procedure` | 계산 절차를 정확히 수행하는가 (예: 이항·정리) |
| **응용 / 문장제** | `application` | 실생활 맥락을 수식으로 옮기는가 |
| **수리적 추론** | `reasoning` | 여러 단계를 논리적으로 연결하는가 |

각 차원에는 `must_preserve` 플래그가 붙는다. **이 플래그가 켜진 차원은 동형 문제에서도 반드시 보존되어야** 한다. 이 플래그가 없으면 "표면상 같은 단원이지만 사실 다른 능력을 측정하는" 거짓 동형이 생긴다.

## 2. 전체 파이프라인 한 눈에

```
┌──────────────────────────────────────────────────────────────────────┐
│  사용자 (학원 강사)                                                    │
│  "중3 이차방정식, 인수분해 문제 3개, 구조 동형"                          │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────────────────┐
│  S1. RAG 검색           (결정론)        packages/agent/.../rag-client │
│  └→ 2,400 건 원본 문제 중 학년·단원·난이도 필터 + 토큰 오버랩 점수      │
│     → RagResult[] (참조 문제 k 개)                                    │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────────────────┐
│  S2. 의도 추출 (Intent)  (LLM 단일)     .../intent-extraction.ts      │
│  └→ 사용자 요청 + 참조 문제 → "어떤 성취기준을 평가하는가" 구조화        │
│     → Intent { achievement_code, evaluation_dimensions[], ... }      │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────────────────┐
│  S3. 문제 생성          (Multi-Agent)   .../problem-generation.ts    │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐            │
│  │ Generator    │ → │ ConstraintCritic│ → │ Refiner       │  ⤴ 최대 3 │
│  └──────────────┘    └──────────────┘    └──────────────┘   loops    │
│  → GeneratedProblem (미검증 — 사용자에게 노출 금지)                    │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────────────────┐
│  S4. SymPy 검증         (결정론 게이트 #1)  packages/math-engine/     │
│  └→ /solve · /verify · /simplify HTTP 호출                            │
│     실패 시 즉시 reject — 사용자에게 도달 금지 (I-V3)                  │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────────────────┐
│  S5. 독립 재풀이        (LLM 단일)      .../independent-resolve.ts   │
│  └→ 다른 프롬프트·온도로 LLM 이 다시 풀이                              │
│     SymPy 답과 불일치 → "주의(warning)" 라벨만 부착                    │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
┌──────────────────────────────────────────────────────────────────────┐
│  S6. 학습목표 매핑      (결정론 게이트 #2) .../objective-mapping.ts  │
│  └→ achievement_code 동일 + must_preserve 차원 보존성 검사            │
│     실패 시 reject (I-V2)                                            │
└─────────────────────────┬────────────────────────────────────────────┘
                          ↓
                   verified 문제만 사용자에게 도달
              (rejected / warning 도 모두 SSE 로 가시화)
```

**4 가지 판정 결과** (`OverallVerdict`):

- `verified` — S4 + S6 모두 통과
- `rejected` — S4 실패 또는 재시도 3 회 초과
- `warning` — S4 통과했으나 S5 가 불일치 (사용자에게 "주의" 라벨로 표시)
- `skipped` — 단계를 건너뛴 경우 (메타데이터에 명시)

## 3. 컴포넌트별 Ablation — 빠지면 무엇이 망가지는가

각 컴포넌트마다 다음 4 가지로 분석한다:

1. **무엇을 하는가** — 코드/스펙 위치와 함께
2. **빠지면 어떻게 되는가** — 구체적 실패 시나리오
3. **현실 예시** — 실제 발생할 수 있는 잘못된 출력
4. **측정 지표** — 어떤 지표가 어떻게 변화할 것인가

---

### 3.1 S1 RAG 제거 시

**무엇을 하는가**
사용자의 학년·단원·난이도 입력을 받아, `openmath_rag_records.jsonl` (2,400 건의 정규화된 원본 문제) 중에서 관련 문제 k 개를 검색한다. 검색 점수는 `achievement_confidence × topic 토큰 오버랩` 의 결정론적 계산. (구현: [`packages/agent/src/tools/rag-client.ts`](../../packages/agent/src/tools/rag-client.ts), 스펙: [`docs/specs/rag-integration.md`](./rag-integration.md), 결정: D-7)

**왜 필요한가**
LLM (예: GPT-4) 의 학습 데이터에는 2022 개정 한국 중등 수학 교육과정의 **구체적 표현** 이 충분히 포함되어 있다고 보장할 수 없다. 성취기준 코드 `9수04-12` 가 어떤 학습 목표를 의미하는지, "인수분해의 인수 쌍 찾기" 라는 표현이 어느 단원에서 정확히 어떤 의미인지를 모를 수 있다. RAG 가 이 격차를 메운다 — **LLM 에게 "이런 문제가 한국 교과서의 정답 표준이다" 라는 참조점을 주는 것**.

**빠지면 어떻게 되는가**
- Intent 추출 단계 (S2) 가 "이 단원이 평가하는 차원이 무엇인지" 자력으로 판단해야 함 → LLM 의 일반 지식에 의존
- 결과: `achievement_code` 가 잘못 매칭되거나, `evaluation_dimensions` 중 `must_preserve` 플래그가 엉뚱한 차원에 붙음
- 따라서 S6 (학습목표 매핑) 의 비교 기준 자체가 흔들리고, **"동형이라고 판정됐지만 사실 다른 단원을 평가하는" 거짓 양성** 이 증가

**현실 예시**
- 입력: "중2 일차함수 단원에서 그래프 해석 문제"
- RAG 없을 때 출력: `2x + y = 5` 의 해 `(x, y)` 를 구하시오 → **이것은 "일차방정식" 평가지 "일차함수 그래프 해석" 평가가 아님**
- RAG 있을 때 출력: 좌표평면 위 두 일차함수 그래프의 교점을 구하고 해석하라 → 평가 차원 일치

**측정 지표**
- **Intent 정확도** (사용자 요청과 추출된 성취기준의 일치율) ↓ 약 30~40% 추정
- **S6 통과율** (학습목표 매핑) ↓ — 잘못된 Intent 가 들어가니 매칭 자체가 안 됨
- **사용자 인지 신뢰도** (이 문제가 정말 내가 가르치는 단원인가?) ↓ 결정적

### 3.2 S2 Intent 추출 제거 시

**무엇을 하는가**
RAG 결과와 사용자 요청을 LLM 에게 보여주고, **"이 문제가 무엇을 평가하는가"** 를 구조화된 JSON 으로 추출한다. 출력은 Zod 스키마(`IntentSchema`) 로 강제 검증된다. (구현: [`packages/agent/src/agents/intent-extraction.ts`](../../packages/agent/src/agents/intent-extraction.ts))

**왜 필요한가**
사용자 입력 ("중3 이차방정식 인수분해") 은 **너무 모호하다.** 같은 단원이라도 다음 두 문제는 다른 능력을 평가한다:
- A. `x² − 5x + 6 = 0` 을 인수분해로 풀어라 → 절차 (procedure)
- B. 두 자연수의 합이 5, 곱이 6 일 때 두 수를 구하라 → 응용 (application) + 추론 (reasoning)

Intent 추출은 **사용자가 명시하지 않은 차원** ("나는 절차만 보고 싶다" vs "응용도 같이 평가하고 싶다") 을 RAG 참조 문제로부터 추론해 명시화한다. 이것이 S6 매핑의 비교 기준이 된다.

**빠지면 어떻게 되는가**
- S3 (생성) 가 "어떤 평가 차원을 의도하는지" 모른 채 문제를 만듦
- S6 (학습목표 매핑) 의 비교 대상이 사라짐 → S6 가 사실상 무의미해짐
- 결과: 같은 단원 내에서도 **무작위 차원의 문제** 가 섞여 나옴

**현실 예시**
- 사용자: "이차함수 응용 문제 3개"
- Intent 없을 때: 세 문제가 각각 그래프 해석, 식 변형, 수직 던지기 (서로 다른 차원) → 학원 강사의 **숙제 의도** 와 어긋남
- Intent 있을 때: 세 문제 모두 `application` 차원 보존 — 일관된 숙제 세트

**측정 지표**
- **세트 내 차원 일관성** (Cronbach's alpha 또는 단순 카테고리 일치율) ↓
- **수업 적합성 평가** (강사 만족도 5점 척도) ↓

### 3.3 S3 Multi-Agent 생성에서 Critic / Refiner 제거 시

**무엇을 하는가**
S3 는 OpenMath 의 유일한 **multi-agent 협력** 단계다 (D-5 ε Hybrid):

```
Generator  → 초안 문제 + 정답 + 풀이 생성
   ↓
ConstraintCritic  → "Intent 의 제약을 어겼는가" 검사
   ↓ (위반 발견 시)
Refiner  → 위반 지점만 수정 (최대 3 회 반복)
```

각 에이전트는 다른 시스템 프롬프트와 온도(temperature)를 사용한다. (구현: [`packages/agent/src/agents/problem-generation.ts`](../../packages/agent/src/agents/problem-generation.ts))

**왜 필요한가**
- Generator 단독이면: LLM 의 **첫 번째 출력** 이 그대로 SymPy 검증으로 직행 → 매번 결과가 들쭉날쭉하고 SymPy 통과율이 낮다
- Critic 이 있으면: "이 문제는 사용자가 명시한 difficulty 를 어겼다", "must_preserve 차원이 다르다" 같은 **제약 위반을 명시적으로 검출** → Refiner 가 타겟팅된 수정 가능

이 패턴은 **debate 와 다르다.** Critic 은 "정답을 다수결로 정하는" 역할이 아니라 **"제약 위반을 검출하는"** 역할이다. 최종 정답 판정은 결정론(S4 SymPy)이 한다. 이것이 D-5 의 핵심 설계 결정이다.

**빠지면 어떻게 되는가**
- Generator 단독 → SymPy 통과율 ↓ 추정 (검증 게이트 통과율이 50→70% 수준에서 80→90% 수준으로 차이)
- 사용자 입장에서 체감: 3 개 요청했는데 1 개만 verified, 2 개는 rejected → 재요청 비용 ↑

**현실 예시**
- 사용자: "쉬운 일차방정식, 정수해만"
- Critic 없을 때: `2x + 3 = 8` (정답 `x = 5/2`, 정수해 제약 위반) → SymPy 는 통과 (방정식 자체는 올바름) → 그러나 **사용자 의도와 다름**
- Critic 있을 때: Generator 가 같은 출력 → Critic 이 "정수해 제약 위반" 검출 → Refiner 가 `2x + 3 = 11` (정답 `x = 4`) 로 수정

**측정 지표**
- **SymPy 게이트 통과율** ↓ 10~20%p 추정
- **사용자 명시 제약 위반율** ↑ 결정적
- **평균 재시도 횟수** ↑ → 응답 latency ↑ + 비용 ↑

### 3.4 S4 SymPy 검증 제거 시 — 가장 치명적

**무엇을 하는가**
[`packages/math-engine/`](../../packages/math-engine/) 는 Python + SymPy + FastAPI 로 구현된 별도 서비스다 (D-2). agent 가 HTTP 로 호출한다:

- `POST /solve` — 방정식 풀이 (해집합 반환)
- `POST /verify` — 두 식의 **기호적 동치성** 검사
- `POST /simplify` — 정규형 (canonical form) 변환
- `POST /differentiate`, `POST /limit` — 미분/극한

LLM 이 제시한 정답을 **기호 계산기로 독립 검증** 한다. 통과 못 하면 즉시 `overall = rejected` (불변식 I-V3).

**왜 필요한가 — D-1 의 핵심**
§1.2 의 MathTrap300 사례. **LLM 단독 검증은 신뢰할 수 없다.** 특히:
- 분수 / 근호 / 다항식 곱 정리
- 음수 / 절댓값 / 부등식 방향
- 다중해 (예: 이차방정식의 두 해를 모두 찾기)

SymPy 는 이런 부분을 결정론적으로 검증한다. **이것이 OpenMath 가 ChatGPT + Wolfram Plugin 류 도구와 본질적으로 다른 점이다** — 후자는 LLM 이 Wolfram 을 *호출할지 말지* 결정하지만, OpenMath 는 **모든 출력이 반드시 SymPy 를 거치는** 강제 게이트다.

**빠지면 어떻게 되는가 — 최악의 시나리오**

이 게이트가 없으면 OpenMath 의 **존재 이유 자체가 사라진다.** 단순히 ChatGPT 한 번 더 호출하는 것과 다를 바 없어진다:

| 빠진 후 가능한 실패 모드 | 빈도 추정 | 영향 |
|---|---|---|
| LLM 이 정답을 틀리게 계산 | 5~15% | 학생이 틀린 답을 정답으로 외움 → 교육 사고 |
| 풀이는 맞지만 정답 표기가 다른 정규형 | ~10% | 학생이 자기 답이 맞는데 틀렸다고 판단 → 좌절 |
| 다중해 중 일부만 제시 | 이차방정식 시 빈번 | 학생이 누락된 해 학습 못함 |
| LLM 이 "검증했다" 고 거짓 보고 | ~5% | 발견 자체가 불가 → 신뢰 무너짐 |

**현실 예시**

생성된 문제: `x² − 5x + 6 = 0` 의 해를 구하시오.

- LLM 이 제시한 정답: `x = 2` (단일 해)
- **SymPy 없을 때:** 통과 → 학생에게 도달 → 학생은 또 다른 해 `x = 3` 을 못 배움
- **SymPy 있을 때:** `/solve` 호출 → 해집합 `{2, 3}` 반환 → LLM 답과 불일치 → `rejected`

**측정 지표**
- **정답 정확도** (생성된 정답이 정말 옳은가) → 신뢰 구간 95% → 본질적으로 불가능 수준으로 하락
- **사용자가 발견한 오답 보고율** ↑ 결정적
- **서비스 신뢰도 / NPS** ↓↓ — 한 번의 오답이 강사의 신뢰를 영구 파괴

> **결론: S4 는 ablation 의 대상이 아니라 시스템의 기둥이다.** S4 제거는 OpenMath 가 아닌 다른 시스템이 된다.

---

### 3.5 S5 독립 재풀이 제거 시

**무엇을 하는가**
S5 는 **다른 시스템 프롬프트와 온도** 로 LLM 에게 문제를 다시 풀게 한다. 단, **정답 판정은 하지 않는다** (D-1). SymPy 답과의 불일치만 기록되어 사용자에게 `warning` 라벨로 표시된다 (불변식 I-V4). (구현: [`packages/agent/src/agents/independent-resolve.ts`](../../packages/agent/src/agents/independent-resolve.ts))

**왜 필요한가**
S4 SymPy 는 "정답이 옳은가" 를 검증하지만, **"문제가 모호하거나 함정인가"** 는 검증하지 못한다. 예:

- 문제: "두 자연수의 합이 5 일 때 두 수를 구하라" → 답이 여러 쌍 (1+4, 2+3) → SymPy 는 "방정식이 아니라서" 검증 불가
- 문제: "다음 식을 간단히 하시오: `x² − 1`" → 정답이 `(x−1)(x+1)` 인지 `x² − 1` 자체인지 모호

이런 경우 **다른 시각의 LLM 풀이** 가 같은 답을 내면 → 신뢰 ↑. 다르면 → 모호함 의심.

이것은 **LLM 의 합의를 정답으로 삼는 게 아니라**, **"불일치 신호" 만 수집해서 사용자에게 라벨링** 하는 방식이다. 정답 판정은 여전히 SymPy(S4)가 한다.

**빠지면 어떻게 되는가**
- S4 가 통과한 문제 중 "모호하거나 다른 해석이 가능한" 문제가 그대로 사용자에게 도달
- 학생/강사가 풀이 중 "이게 정말 맞나?" 의심하지만 OpenMath 는 아무 신호도 안 줌
- **silent failure** — 검출 불가능한 실패

**현실 예시**
- 문제: `\sqrt{16}` 의 값은?
- LLM 정답: `4`
- SymPy 검증: `/verify(sqrt(16), 4)` → True → 통과
- S5 재풀이 LLM 출력: `±4` (음수 제곱근까지 고려)
- 결과: `warning` 라벨 → 강사가 "이 문제는 학생 수준에 따라 다르게 받아들여질 수 있음" 인지

**측정 지표**
- **모호한 문제 검출율** (인간 평가 기준) → S5 제거 시 0% 로 수렴
- **사용자가 발견한 모호함 보고율** ↑
- **warning 비율** ≈ 5~10% 추정 — 이 만큼이 silent failure 로 변환됨

---

### 3.6 S6 학습목표 매핑 제거 시

**무엇을 하는가**
S2 에서 추출된 Intent 의 `achievement_code` 와 `evaluation_dimensions` 를, S3 에서 생성된 문제의 `inferred_intent` 와 비교한다. **결정론 매칭** 이 기본이고, `mode == "conceptual"` 일 때만 LLM 보조 (그러나 최종 판정은 결정론). (구현: [`packages/agent/src/agents/objective-mapping.ts`](../../packages/agent/src/agents/objective-mapping.ts))

**검증 항목:**
1. `achievement_code` 가 동일한가
2. `evaluation_dimensions` 중 `must_preserve: true` 인 차원이 모두 보존되었는가
3. `required_techniques` 가 모두 포함되어 있는가

**왜 필요한가**
S4 SymPy 가 "정답이 옳은가" 만 검증하기 때문에, **"옳은 정답인데 잘못된 평가 차원" 인 문제** 는 S4 를 통과한다.

예: 사용자가 "절댓값 방정식 (개념 이해)" 을 요청 → LLM 이 `|x| = 3` 을 만듦 → SymPy 통과 → **그러나 이것은 "개념 이해" 가 아니라 "단순 절차"** 평가. S6 가 없으면 이런 문제가 통과한다.

**빠지면 어떻게 되는가**
- 같은 단원 내에서도 **다른 차원** 의 문제가 verified 로 표시됨
- Intent 추출(S2)의 의미가 사라짐 (어차피 비교 대상이 없으니)
- "동형" 의 정의가 무너짐 → 시스템은 그냥 "그럴듯한 문제 생성기"

**현실 예시**
- 사용자 요청: 중3 이차방정식 단원, `concept` 차원 `must_preserve`
- S3 생성: "`x² − 7x + 12 = 0` 을 풀어라" → SymPy 통과 (정답 `{3, 4}`)
- S6 없을 때: verified → 사용자에게 도달 → 그러나 이것은 절차 평가
- S6 있을 때: `evaluation_dimensions` 비교 → `concept` 차원이 평가 안 됨 → reject 또는 재시도

**측정 지표**
- **차원 보존율** (Intent vs Inferred Intent 의 차원 일치율) → S6 제거 시 LLM 운에 맡김
- **거짓 동형 비율** ↑ — 표면상 같지만 평가가 다른 문제

## 4. "왜 Multi-Agent Debate 가 아니라 결정론 게이트인가" — 대안 아키텍처와의 비교

이 시스템을 처음 보는 사람은 **"왜 더 멋진 multi-agent debate / consensus / parliament 방식이 아닌가?"** 라고 물을 수 있다. 학계에 다음과 같은 인기 패턴이 있다:

### 4.1 대안 A — Multi-Agent Debate (e.g., "Society of Minds")

여러 LLM 에이전트가 각자 문제를 풀고, 서로의 답을 비판하며 합의에 도달하는 방식.

**왜 OpenMath 가 채택하지 않았는가** ([`docs/specs/architecture.md`](./architecture.md) D-5 명시):

> **"합의 ≠ 정답."** Debate 의 합의는 *다수결 LLM 신뢰* 위에 선다. LLM 끼리 같은 편향을 공유하면 모두 같은 오답에 합의할 수 있다. 수학 검증에서 이것은 D-1 의 정면 위반이다.

**구체적 실패 시나리오:**
- 3 개 LLM 에이전트가 모두 `\sqrt{16} = 4` 라고 답함 → 합의 → 정답으로 채택
- 그러나 `±4` 가 진짜 답
- SymPy 가 없으니 이 오류를 잡을 수단이 없음

### 4.2 대안 B — Self-Consistency (단일 모델, 다중 sampling)

같은 LLM 을 여러 번 호출해서 답이 가장 많이 나온 것을 채택.

**왜 채택하지 않았는가:**
- 비용이 N 배 (5 회 sampling = 5 배 LLM 호출)
- 모델이 특정 오답에 편향되어 있으면 "다수결" 이 무의미
- 결정론적 보장 없음 — 같은 입력에 같은 출력 보장 안 됨

### 4.3 대안 C — Single LLM + Wolfram Plugin

ChatGPT 가 Wolfram 을 *필요할 때* 호출.

**왜 채택하지 않았는가:**
- LLM 이 "호출할지 말지" 를 자율 결정 → **검증 누락 가능성**
- 비결정론적 — 같은 문제에 어떤 때는 검증, 어떤 때는 미검증
- 캡스톤/논문 수준에서 **"검증된 문제만 도달한다" 라는 보장** 을 줄 수 없음

### 4.4 OpenMath 의 채택 — ε Hybrid

D-5 ε (Hybrid) 선택의 의미:

- **"진짜 multi-agent"** — S3 의 Generator-Critic-Refiner 는 실제로 다른 모델/프롬프트로 협력
- **"결정론 게이트"** — S4, S6 는 LLM 의 의견을 묻지 않고 코드로 판정
- **"LLM 보조는 명시적으로 분리"** — S5 는 LLM 이 의견을 내지만 *판정자가 아니라 신호 제공자* 라는 역할이 코드 구조에 박혀 있음

이 분리가 없으면 — 즉 한 LLM 이 "내가 만들고 내가 검증한다" 면 — 그 검증을 어떻게 신뢰할 것인가? 이것이 OpenMath 의 핵심 차별화이고, **각 단계 ablation 의 의미가 살아나는 이유** 다.

## 5. 정량적 검증 실험 설계 (논문 / 발표용)

### 5.1 실험 변형 (variants)

| 코드 | 구성 | 비교 의미 |
|---|---|---|
| `V0` | Full Pipeline | 베이스라인 |
| `V1` | − S1 RAG | RAG 의 기여 |
| `V2` | − S2 Intent | 의도 추출의 기여 |
| `V3` | − S3 Critic/Refiner (Generator only) | Multi-agent 의 기여 |
| `V4` | − S4 SymPy (LLM 검증으로 대체) | **결정론 검증의 기여 (가장 중요)** |
| `V5` | − S5 재풀이 | 모호함 검출의 기여 |
| `V6` | − S6 학습목표 매핑 | 동형성 보장의 기여 |
| `V7` | Single LLM only (모든 단계 제거) | "ChatGPT 한 번" 베이스라인 |

### 5.2 측정 지표

| 지표 | 측정 방법 | 의의 |
|---|---|---|
| **정답 정확도** | 생성된 문제 100 개에 대해 SymPy 독립 평가 | 핵심 — 잘못된 정답 비율 |
| **동형성 유지율** | 인간 평가자 3 명이 "같은 평가 차원인가" 5점 척도 | Intent / S6 의 기여 |
| **단원 적합성** | 강사 평가자가 "이게 정말 [중3 인수분해] 인가" Y/N | RAG / S6 의 기여 |
| **모호함 발견율** | 평가자가 "이 문제 모호하다" 라고 표시한 비율 | S5 의 기여 |
| **재시도 횟수** | 평균 attempt_count | Critic / Refiner 효율성 |
| **사용자 신뢰도** | 강사 5명 인터뷰, NPS | 통합 효과 |

### 5.3 데이터셋

- **요청 100 건**: 학년(중1/2/3) × 단원(5 개씩) × 모드(structural/conceptual) × 차원(4 개) 조합 샘플링
- **인간 평가자**: 수학 교사 / 강사 3 명 + 학부생 평가자 2 명
- **블라인드 평가**: 평가자는 어느 variant 출력인지 모름

### 5.4 가설

| 가설 | 예측 |
|---|---|
| H1 | V0 의 정답 정확도 > V4 의 정답 정확도 (큰 차이, p < 0.01) |
| H2 | V0 의 동형성 유지율 > V6 (중간 차이) |
| H3 | V0 의 단원 적합성 > V1 (RAG 없이 LLM 일반 지식만으로는 부족) |
| H4 | V3 (Generator only) 의 평균 재시도 > V0 (Critic 있는 쪽이 효율적) |
| H5 | V7 (Single LLM) 은 모든 지표에서 V0 대비 통계적 유의 하락 |

### 5.5 결과 보고 형식

표로 V0 ~ V7 의 각 지표를 비교. 핵심 발견 사항을 **delta** (V0 대비 변화량) 로 시각화. 95% 신뢰 구간 포함.

## 6. 결론

### 6.1 핵심 메시지

OpenMath 는 **"LLM 단독 = 동형 문제 생성 불가능"** 이라는 가설 위에 설계된 시스템이다. 6 단계 파이프라인의 각 컴포넌트는:

1. **S1 RAG** — LLM 의 일반 지식과 한국 교육과정 간 격차를 메움
2. **S2 Intent** — 사용자의 모호한 요청을 명시적 평가 차원으로 구조화
3. **S3 Multi-Agent** — 제약 위반을 검출하고 타겟팅된 수정 (debate 아님)
4. **S4 SymPy** — *결정론* 정답 검증 (대체 불가능한 핵심)
5. **S5 재풀이** — 모호함 / silent failure 신호 수집
6. **S6 학습목표 매핑** — "옳은 정답이지만 잘못된 평가 차원" 검출

각 단계가 빠지면 위에서 분석한 구체적 실패 모드가 발생한다. 이 중 어느 것도 ChatGPT 한 번 호출로는 해결되지 않는다.

### 6.2 "왜 ChatGPT 가 아닌가" — 한 줄 답

> **ChatGPT 는 "그럴듯한 문제" 를 생성하지만, OpenMath 는 "검증된 문제만 도달하는 것" 을 보장한다.**

이 보장은 **결정론 게이트(D-1)** 가 있어야 가능하고, 결정론 게이트가 유의미하려면 **LLM 생성과 검증의 책임 분리(D-5 ε Hybrid)** 가 있어야 한다. 두 결정이 OpenMath 아키텍처의 척추다.

### 6.3 한계 및 후속 연구

- **현재 RAG 는 토큰 오버랩** — 의미론적 임베딩(vector search) 도입 시 성능 향상 여지
- **2,400 건 데이터 한정** — 1만건+ 확장 시 retrieval 품질 변화 측정 필요
- **Q-5 의 재시도 ≤ 3 회 제한** 은 잠정 — 비용 vs 통과율 trade-off 후속 실험 필요
- **인간 평가자 수 부족** — 정식 논문에서는 평가자 간 신뢰도(Cohen's kappa) 보고 필요

## 부록 B — 핵심 불변식 (Invariants)

검증 결과의 일관성을 보장하는 5 개 불변식 ([`verification.schema.ts`](../../packages/agent/src/schemas/verification.schema.ts) 참조):

| 코드 | 내용 |
|---|---|
| **I-V1** | 단계를 건너뛴 경우 `skipped` 메타데이터에 명시 |
| **I-V2** | `overall == "verified"` ⟹ `sympy_verify == "passed"` AND `objective_map == "passed"` |
| **I-V3** | `sympy_verify == "failed"` ⟹ `overall ≠ "verified"` (D-1 의 핵심 코드화) |
| **I-V4** | `overall == "warning"` ⟺ (`sympy_verify == "passed"` AND `re_solve == "failed"`) |
| **I-V5** | `attempt_count > 3` ⟹ `overall == "rejected"` (Q-5 잠정 비용 제어) |

이 불변식들은 Zod 스키마로 강제되어, **"sympy failed 인데 verified" 같은 논리적 모순** 이 코드 레벨에서 차단된다.
