0 먼저 이해하기 — stub이 뭔가?
코드가 있는데 왜 안 돌아가는지 이해하는 게 핵심

stub(스텁)이란?

코드 파일은 있고 함수 이름도 있는데, 안에 실제 내용이 없고 '나중에 만들어줘' 라고 적어둔 상태야. 마치 건물 외벽과 뼈대는 다 있는데 내부 공사가 하나도 안 된 것처럼.

stub 코드 예시 함수 이름: createGeneratorAgent() 내부 내용: throw new Error('not implemented yet') → 호출하면 즉시 오류 발생. 아무것도 안 함.

2 발견된 문제 — 환경변수 이름 불일치
.env.example 에 적힌 이름과 실제 코드가 인식하는 이름이 달라서 조용히 무시됨

.env.example 에 적힌 이름코드가 실제로 읽는 이름문제

CLIPROXY_BASE_URLLLM_BASE_URL이름이 달라 → 조용히 무시됨
CLIPROXY_API_KEYLLM_API_KEY이름이 달라 → 조용히 무시됨
LLM_PROVIDER=cliproxyopenai-compatible 써야 함'cliproxy' 값 자체가 거부됨
CLIPROXY_MODEL코드에 정의 없음이 변수 자체가 코드에 없음
올바른 .env 설정 LLM_PROVIDER=openai-compatible LLM_BASE_URL=http://127.0.0.1:8317/v1 LLM_API_KEY=my-secret-key MATH_ENGINE_URL=http://localhost:8000

🔴 약점 1 — RAG 검색 엔진의 Zod 검증 누락 (D-7 부분 위반)
위치: rag-client.ts:148-182 (toIndexedProblem 함수)
현상: JSONL 기출문제를 메모리에 로드할 때 TypeScript 타입 강제만 있고, Zod 런타임 검증 (SourceProblemSchema.parse) 호출 없음.
결과: 깨진 데이터가 들어와도 통과. 예: grade: "abc" (숫자여야 함) 또는 grade: null 도 그대로 통과 → I-S1~I-S5 (SourceProblem 불변식 5건) 강제 안 됨.
권장: SourceProblemSchema.parse(problem) 한 줄 추가.
🔴 약점 2 — health endpoint 가 가짜 OK 반환 (D-3 위반)
위치: server/routes/health.ts:10
현상: health 핸들러가 _mathEngine 파라미터를 받아놓고 한 번도 호출하지 않음. 응답은 항상 {status:"ok"}.
결과: math-engine 서버가 죽어도 agent /health 는 200 OK 반환 → 모니터링이 silent failure 를 못 잡음.
권장: await mathEngine.health() 호출 + 실패 시 503 반환.
🔴 약점 3 — Windows 에서 main() 호출 안 됨
위치: src/index.ts:14
현상: 진입점 가드 if (import.meta.url === `file://${process.argv[1]}`) 가 Windows 에서 백슬래시/슬래시 차이로 매번 false. main() 자체가 호출 안 됨.
결과: pnpm dev 가 침묵 (에러도 출력 안 됨). 개발자가 "내가 뭘 잘못했나" 헷갈림.
권장: pathToFileURL(process.argv[1]).href 변환 후 비교.
🟡 약점 4 — 모델 ID 하드코딩 (cliproxy 즉시 fail)
위치: config/models.ts (DEFAULT_MODELS)
현상: 6개 agent 역할의 모델이 전부 gpt-4o / gpt-4o-mini 하드코딩. env 변수로 override 불가능.
결과: cliproxy 모델 목록에 gpt-4o 없음 (cliproxy 는 claude-haiku-4-5, gemini-2.5-flash 등 제공) → 즉시 unknown provider for model gpt-4o 에러.
권장: env 변수 LLM_MODEL_* 로 override 가능하게 변경.
요약: stub 17건 채우는 작업과 별개로, 이미 동작하는 코드 4건에도 fix 필요. 약점 1, 2 는 D-7/D-3 도메인 원칙 위반이라 우선순위 높음.
